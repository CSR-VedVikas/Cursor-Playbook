CURSOR · THE BASICS
Getting Started with Cursor
Cursor is a code editor with AI built into it. You describe what you want in plain language, and it reads your whole project to help you write and fix code.

What Is Cursor

With most AI tools you copy code into a chat window and paste the result back. Cursor's AI lives inside the editor instead. It reads your whole project, understands how the files connect, and helps you build software in plain English. Cursor is built on the same foundation as VS Code, so if you already use it, your extensions, themes, and shortcuts carry over in one click. If you have never used a code editor, that is fine too.

Most AI coding tools are plugins added on top of an editor. In Cursor the AI is part of the editor itself. When you ask a question, it already knows the contents of every file. When it writes code, it puts it in the right file and the right place, without you navigating there.

The Four Ways to Interact

Each mode has a different scope. Learn what each one is for before you start.

The Agent takes a task described in plain English, works out what needs to happen, and carries it out across your whole project: reading files, writing code, running commands, and searching the web.

Writing Good Prompts

What the Agent builds depends on how clearly you describe it. A strong prompt covers four things:

  • What you want built — the feature or change, described specifically
  • The inputs and outputs — what goes in, what should come out
  • Constraints — what it should not do, or rules it must follow
  • An example or reference — a file, a pattern, or a sample to follow

Debugging (Debug Mode)

A dedicated agent loop for bugs that are hard to reproduce. Rather than guessing, it instruments your running app with logs, observes what actually happens, forms hypotheses, and asks you to verify before applying a fix. Activate with /debug. The loop: you describe the bug & how to reproduce → the Agent adds runtime logs → you run and reproduce → the Agent reads the logs and forms hypotheses → it proposes a fix with reasoning → you verify or give feedback → repeat until resolved.

EXAMPLE DEBUG PROMPT
When I call extract_features('') the function returns None instead
of raising a ValueError. To reproduce: run url_checker.py and call
extract_features('') from the Python shell. I have already confirmed
the input-validation block runs. The issue appears to be in the
early-return logic.

Reviewing Agent Changes

After the Agent finishes you enter the review phase — this is where you decide what actually gets kept. In the diff, green = added, red = removed/replaced, white = unchanged context.

CheckWhat to look for

💡Tip: After accepting a change, run your project before moving on. Catching problems early is far faster than untangling several accepted changes later. Use Cmd/Ctrl+Z to undo AI-made changes individually.

Tools the Agent Can Call

The Agent chooses tools based on the task; you do not pick them manually. Terminal: runs commands (installing packages, running your app, tests) — by default it asks approval first. Browser: a built-in sidebar to view your running app and check layouts. Search: semantic search (by meaning) and grep (exact text) across the project. Canvases: a visual workspace to plan and diagram alongside code. Worktrees: run multiple Git branches at once in separate workspaces.

⚠️Note: Always read terminal commands before approving — a misunderstood instruction could run a command you did not intend.

Customising Cursor
Cursor works for any project out of the box. You get more out of it by telling the Agent about your project, connecting it to your tools, and saving repeatable workflows so you never explain the same thing twice.
💡Recommended order for beginners: Start with Rules, then MCP, then Plugins. Skills, Subagents, and Hooks are for advanced workflows and can wait.

Rules — the Most Important Customization

Rules are standing instructions the Agent reads at the start of every conversation: your coding standards, project conventions, preferred tools, and constraints. Language models have no memory between chats, so Rules re-supply your instructions automatically each session. The current format stores them as Markdown files inside a .cursor/rules/ folder, one file per topic.

PROJECT LAYOUT WITH RULES
your-project/
├── .cursor/
│   └── rules/
│       ├── index.mdc      <- project overview (always applied)
│       ├── python.mdc     <- Python language conventions
│       ├── security.mdc   <- security standards
│       └── testing.mdc    <- testing guidelines
├── app.py
└── requirements.txt
Rule typeWhen it appliesBest used for
Always (alwaysApply: true)Every single conversationProject overview, critical constraints
Auto Attached (globs: ["*.py"])When a matching file is open or mentionedLanguage-specific rules, file-type standards
Agent RequestedWhen the Agent decides it's relevantSpecialised, domain-specific workflows
ManualOnly when you reference it with @rulenameRules for rare edge cases

CURSOR · PRO TIPS
Pro Tips & Community Wisdom
The habits that separate beginners who get inconsistent results from experienced users who ship reliably. These come from the official guides and the Cursor community.

Give the Agent the Right Context

The fastest way to improve output is to give the Agent the right context before it starts. Type @codebase to trigger a semantic search across every indexed file (verify indexing under Settings > Indexing; Reindex if results feel wrong). Tag a whole directory with @folder when a task needs a broad view of one module. Add external library docs under Settings > Features > Docs, then reference them with @docs so the Agent uses live content instead of stale training data.

💡Index the docs for every library in your requirements.txt. After that, the Agent stops suggesting deprecated patterns from older versions of the same library.

Ten Pro Agent Tips

  • Plan Mode before big tasks. Shift+Tab before anything touching more than 2–3 files — eliminates most "the Agent built something completely different" problems.
  • Right-click → Add to Chat, or drag files from the explorer into the chat — the fastest way to pin context.
  • Custom commands for repeated prompts. Save a prompt structure once under Settings > Custom Commands.
  • Paste images directly into chat — a UI screenshot, a whiteboard photo, or an error dialog. The Agent reads the image.
  • Duplicate a chat to try a different approach. The original stays as a safe fallback; run both in parallel.
  • Check context visibility before running. Click the context eye icon to see exactly what files, rules, and docs are loaded; add what's missing, remove what's irrelevant.
  • Track usage from the status bar. Switch to lighter models (mini / Haiku) for simple tasks like renaming or reformatting.
  • Learn three shortcuts: Cmd+I (Agent), Cmd+L (Chat), Cmd+K (inline edit). Cmd+/ switches model; Cmd+N starts a new chat.
  • Start fresh chats for each task. The Agent re-reads the whole thread every response, so a 30-message thread costs more and gives muddier results than a focused 5-message one.
  • Use checkpoints to roll back. Cursor saves points before and after each Agent action — like git commits for your conversation. Roll back if it goes the wrong way.

Productivity Wins to Use Immediately

Enable Codebase Indexing on every project (Settings > Features) — the Agent can only reason about files it has indexed. Use Cmd+K for surgical edits when you know exactly which block to change — faster and more precise than Chat. And accept suggestions incrementally, not all at once: review the diff file by file, accept what you are confident about, and address the rest with a follow-up.

⚠️Note: Mass-accepting Agent output on a large refactor is one of the most common beginner mistakes. The diff review is where you catch logic errors that would not show up until runtime.

Community & GitHub Wisdom

Always commit before running the Agent on anything substantial. If the result is wrong, git checkout gets you back instantly.

SAFETY NET

# ...run the Agent...
git checkout .   # discard all agent changes immediately

CellScope Chat

Tumor analysis assistant — ask anything about the Breast Cancer Wisconsin dataset in plain English
The Breast Cancer Wisconsin dataset contains 569 tumor samples. Each sample comes from a Fine Needle Aspiration biopsy — a procedure where a thin needle extracts cells from a breast mass. The cells are digitized and analyzed, producing 30 numeric measurements per sample: radius, texture, perimeter, area, smoothness, compactness, concavity, concave points, symmetry, and fractal dimension — each measured as mean, standard error, and worst value. Every sample is labelled benign or malignant.

The problem: most people who encounter these numbers have no idea what they mean. Is a radius_mean of 18.5 normal? Is a concavity_mean of 0.15 typical for benign cases?

CellScope Chat is a conversational assistant that handles four types of questions: predictions (give it measurements, it classifies and explains), feature explanations (plain English definitions), statistics (averages and ranges from the dataset), and comparisons (how your values compare to typical cases). The Logistic Regression model handles classification. Ollama handles natural language understanding. A permanent disclaimer banner makes the educational scope clear at all times.
🧠 ML Concepts Used in CellScope
Logistic Regression headline model
A classification algorithm that predicts which of two categories something belongs to — here, benign or malignant. It learns a weight for each of the 30 measurements and combines them into a single value between 0 and 1. CellScope uses it because the task is a clean two-class decision, and Logistic Regression is fast, reliable, and easy to explain — you can see exactly which measurements pushed the result toward malignant.
Train / Test Split 80 / 20
Before training, the 569 samples are split — 80% to teach the model, 20% held back to test it. The model never sees the test portion during training, so measuring accuracy on that held-back data shows how well it performs on samples it has never encountered. This is how CellScope confirms its ~97% accuracy is genuine and not just memorisation.
Confidence & Probability predict_proba()
Logistic Regression doesn't just say "malignant" — it outputs a probability (e.g. 0.87). CellScope turns that into the confidence percentage shown next to every prediction ("87% confident: Benign"). A value near 50% means the model is unsure; a value near 100% means it is very sure. Being honest about uncertainty matters for an educational tool.

Luxury property valuation assistant — describe a house in plain English, get a price estimate in Indian rupees with an explanation
✦ This project uses a Kaggle dataset downloaded automatically via kagglehub — no manual CSV download required. Ollama handles conversational replies while regex handles feature extraction for reliability.
The Housing Prices dataset contains property records with 12 features: area (sq ft), bedrooms, bathrooms, stories, parking, and seven yes/no amenity flags (mainroad, guestroom, basement, hotwaterheating, airconditioning, prefarea) plus furnishingstatus. The Random Forest model handles the prediction. Ollama handles conversational replies. Regex handles feature extraction from natural language.
🧠 ML Concept Used in HomePricer
Random Forest Regression headline model
A regression model predicts a number rather than a category — here, the price of a house. A Random Forest builds many separate decision trees, each making its own price guess, then averages them into one final estimate. Averaging many trees cancels out the mistakes of any single tree, so the prediction is more stable and accurate. HomePricer uses it because house price depends on many features interacting in non-linear ways (a pool matters more in a big house than a small one), and Random Forest captures those interactions well — while still letting you see which features pushed the price up or down.

CardioCheck Chat

Heart disease risk assistant — ask about real lifestyle factors (age, cholesterol, chest pain, max heart rate) in plain English, not abstract cell measurements
✦ Cleanest project for absolute beginners — every feature is something a person already understands (your age, your cholesterol, your chest pain). The build is the same FastAPI + Ollama shape as CellScope, so if you finished CellScope, this one will feel familiar. Running in about two hours.
The UCI Heart Disease dataset (Cleveland subset) contains 303 patient records. Each record has 13 features describing one person: age, sex, chest pain type, resting blood pressure, cholesterol, fasting blood sugar, resting ECG result, maximum heart rate achieved, exercise-induced angina, ST depression (oldpeak), ST segment slope, number of major vessels seen on fluoroscopy, and thalassemia result. Every record is labelled heart disease present (1) or not present (0).

Why this dataset for beginners: unlike CellScope's 30 abstract cell-nucleus measurements, these are features a normal person can relate to. Someone can read "cholesterol 270, max heart rate 145" and have an intuition for it. That makes it the easiest dataset to demo to a non-coder — the gap between "numbers in a CSV" and "things about a real person" almost disappears.

CardioCheck Chat handles the same four question types as CellScope: predictions (give it a few clinical values, it classifies and explains), feature explanations (what is "chest pain type", in plain English), statistics (averages and ranges from the 303 patients), and comparisons (how one value relates to typical cases). The Logistic Regression model does all classification. Ollama does all natural-language understanding. A permanent disclaimer banner — "Educational only · not a diagnosis · not medical advice" — stays fixed on screen at all times, because this dataset invites people to type their own numbers.
🧠 ML Concepts Used in CardioCheck
Logistic Regression headline model
A classification algorithm that predicts which of two categories something belongs to — here, heart disease present or not present. It learns a weight for each of the 13 features and combines them into a single value between 0 and 1. CardioCheck uses it because the task is a clean two-class decision, and Logistic Regression is fast, reliable, and — crucially for teaching — transparent: you can read off which features pushed the result toward "disease present" and explain it in one sentence.
Feature Scaling StandardScaler
The 13 features live on wildly different scales — cholesterol is in the hundreds (e.g. 270), oldpeak is a tiny decimal (e.g. 1.4), sex is just 0 or 1. Without scaling, the big numbers would dominate just because they're big. StandardScaler rescales every feature to the same footing before training, so the model weighs them fairly. The same scaler is saved and reused at prediction time — never re-fit on a single user's input.
Train / Test Split 80 / 20
The 303 records are split — 80% to teach the model, 20% held back to test it. The model never sees the test portion during training, so the accuracy measured on that held-back slice (typically above 80% for this dataset) reflects performance on patients it has never encountered, not memorisation.
Confidence & Probability predict_proba()
Logistic Regression doesn't just say "disease" — it outputs a probability (e.g. 0.78). CardioCheck shows that as the confidence percentage next to every prediction. A value near 50% means the model is genuinely unsure; being honest about that uncertainty is the whole point of an educational tool that handles health-adjacent numbers.

Lab 1 · Gradient Descent — One Engine, Many Models

Almost every model your projects use is trained the same way. You have a model (a formula with adjustable numbers called parameters), a loss (one number saying how wrong it currently is), and gradient descent (a rule that nudges the parameters downhill on that loss). Repeat the nudge a few hundred times and the model "learns".

The gradient ∇L is the direction of steepest increase in the loss, so we step in the opposite direction: θ ← θ − η·∇L. The learning rate η is the step size — too small and it crawls, too large and it overshoots and diverges. The big idea this lab makes visible: switch the model and the loss, and the exact same descent loop still trains it.

Lab 2 · From Pixels to Prediction — How a CNN Sees

To a computer an image is just a grid of numbers — each pixel a brightness from 0 to 1. A plain neural network can already learn to read these numbers and recognise digits: feed in the brightness values, stack a few layers, and with enough examples each layer learns to pick up a useful piece — edges, then loops, then whole digits.

But a fully-connected network treats every pixel as unrelated and needs a separate weight for each one — millions of them, with no sense of "nearby". A convolutional layer fixes both problems with one idea: slide a tiny grid of weights (a kernel) across the image, reusing the same weights everywhere. That gives it spatial awareness and a tiny parameter count. Below, you are the convolution — paint an image, pick a kernel, and watch the feature map form.

Why stack layers — the feature hierarchy

One conv layer finds simple patterns. Feed its output into another, and the next layer combines those into richer ones. Depth is what lets a CNN go from edges to objects.

  5. ReLU & pooling. Pick Edge, Slide once, then tick ReLU (negatives snap to 0); toggle Max vs Average pooling — the map halves, Max keeps the strongest activations.
  6. Train & fool the classifier. Press Load & train, wait for "ready", and draw a 7 large and centred → high confidence. Now draw it tiny in a corner → confidence drops. That is why centring and scaling matter.

Lab 3 · Upload & Read — OCR on Any Image

The two labs above read a single digit you draw on a fixed grid. This one is the opposite: hand it a photo or screenshot with printed digits or letters sitting anywhere, and it finds every character and reads it back. That job is called OCR — optical character recognition, the same trick behind scanning a document, a receipt, or a number plate.

🌐 Heads up — this lab needs the internet. Every other section of this playbook runs fully offline. The first time you press Scan, it downloads a small OCR engine (Tesseract.js); that takes roughly 10–30 seconds, then it is cached so later scans are quick. Your image is read entirely inside your browser — it is never uploaded to any server. Clean printed text reads reliably; small, blurry, or handwritten characters are hit-or-miss.

How it works

      
OCR is two jobs stitched together: first decide where the text is, then decide what each mark says. That is why characters can sit anywhere on the image and still get read.

      

        
StageWhat happensWhy it matters

        
          
Page segmentationThe engine scans the whole image and boxes off the regions that look like text, line by line and word by word.This is why your digits or letters can be placed anywhere — they do not need to be on a grid.

          
RecognitionEach boxed glyph is matched against the shapes the model learned for every letter and digit.Turns pixels into actual characters you can copy, search, or feed into code.

          
ConfidenceEvery word gets a 0–100 score for how sure the model is.Low scores flag shaky reads — shown here as amber boxes so you know what to double-check.

          
Layout rebuildThe reader knows each word's position, so it rebuilds spacing from the gaps between words and line breaks from their vertical positions.Your indentation, spacing, and blank lines survive the scan instead of collapsing into one run-on block.

          
Print vs handwritingThe model was trained mostly on printed fonts.Clean print scores high; handwriting, tiny text, and blur often misread or get skipped.
